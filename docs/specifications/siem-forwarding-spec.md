## 1. Overview

The **SIEM Forwarder** is the bridge between CI/CDecoy's event pipeline and an organization's existing security operations tooling. It subscribes to events on NATS, reshapes them into the target SIEM's expected format, and delivers them over the appropriate transport.

The forwarder operates in one of two modes:

**Normalized mode** (`FORWARDER_MODE=normalized`) subscribes directly to the adapter event stream â€” the raw, schema-conformant events produced by adapters before any enrichment occurs. This mode is for teams that either don't run the CI/CDecoy CTI pipeline (they do their own enrichment in the SIEM) or want the lowest-latency path from decoy interaction to SIEM alert. Events in this mode contain the Common Event Schema fields defined in the Adapter Contract Specification. They do not contain GeoIP, MITRE ATT&CK mappings, or threat feed data.

**Enriched mode** (`FORWARDER_MODE=enriched`) subscribes to the output of the CTI enrichment pipeline â€” events that have been GeoIP-resolved, ATT&CK-mapped, threat-feed-correlated, and severity-reclassified. This mode produces the richest SIEM events and is the default for teams running the full CI/CDecoy stack.

Both modes share the same formatter and output module code. The difference is which NATS streams the forwarder subscribes to and whether enrichment fields are present in the event payload. Formatters handle missing enrichment fields gracefully â€” they omit the corresponding output fields rather than erroring.

## 3. Contract Summary

The SIEM Forwarder MUST:

1. Subscribe to the configured JetStream streams via durable pull consumers based on `FORWARDER_MODE`.
2. Deserialize each message as a CI/CDecoy event payload â€” either Common Event Schema (normalized mode) or enriched variant (enriched mode).
3. Transform the event into the configured output format (JSON, CEF, LEEF, or ECS).
4. Handle missing enrichment fields gracefully: omit the corresponding output fields rather than erroring or inserting placeholders.
5. Deliver the formatted event to the configured SIEM target.
6. ACK the NATS message only after successful delivery confirmation.
7. NAK on delivery failure, allowing JetStream to redeliver according to its retry policy.
8. Expose health and delivery metrics, including the active mode.

The SIEM Forwarder MUST NOT:

1. Modify, enrich, or filter events based on content. If the operator wants enrichment, they run the CTI pipeline and use enriched mode.
2. Perform deduplication beyond what NATS JetStream provides natively.
3. Buffer events to disk. In-memory buffering for batched delivery is acceptable; persistence is NATS's responsibility.
4. Reject or error on events missing enrichment fields. The Common Event Schema required fields (defined in the Adapter Contract Specification) are the only hard requirements.

## 5. Event Shapes by Mode

The forwarder must handle two distinct event shapes depending on mode. Both share the Common Event Schema base. The enriched shape adds an `enrichment` object.

### 5.1 Normalized Event (Common Event Schema)

This is the event shape produced by adapters, as defined in the Adapter Contract Specification. It contains only the fields the adapter can determine from the honeypot's native output.

```json
{
  "event_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "timestamp": "2026-04-02T18:47:33.219Z",
  "version": "1.0",
  "source": {
    "decoy": "bastion-dmz-01",
    "tier": 3,
    "pod": "decoy-bastion-dmz-01-7f8b9c-x4k2",
    "node": "k3s-worker-02"
  },
  "session_id": "cowrie-a1b2c3d4",
  "event_type": "command.exec",
  "source_ip": "45.33.32.156",
  "source_port": 48932,
  "severity": "medium",
  "data": {
    "command": "cat /etc/shadow",
    "cwd": "/root",
    "username": "root"
  },
  "_adapter": {
    "name": "cowrie",
    "version": "0.2.0",
    "original_event_type": "cowrie.command.input",
    "ingest_latency_ms": 12
  }
}
```

**What's absent:** No `enrichment` object. No GeoIP, no MITRE mappings, no threat feed scores. Severity is the adapter's baseline assessment, not the pipeline's reclassification.

### 5.2 Enriched Event

This is the event shape produced by the CTI enrichment pipeline. It wraps the normalized event and adds an `enrichment` object.

```json

    "tier": 3,

    "node": "k3s-worker-02"

    },

        {
          "id": "T1003.008",

          "tactic": "credential-access"
        }
```

### 5.3 Enrichment Field Presence

Formatters MUST treat every field inside `enrichment` as optional, even in enriched mode. The enrichment pipeline may not have data for every field â€” GeoIP may fail for a private IP, a new source IP may not appear in any threat feed, and not every command maps to a MITRE technique. The rule is: if the field exists, format it; if it doesn't, omit the corresponding output field silently.

This means the same formatter code works for both modes. In normalized mode, the entire `enrichment` object is absent. In enriched mode, individual enrichment fields may be absent. The formatter's behavior is identical in both cases â€” it checks for field presence and omits what isn't there.

## 6. Output Formats

Each formatter takes a CI/CDecoy event (normalized or enriched) and produces a byte payload suitable for the target SIEM. The formatter does not control delivery â€” it produces bytes, and the output module ships them.

### 6.1 JSON (Passthrough)

The event is serialized as-is â€” whatever fields are present on the event object are included. In normalized mode, this produces the Common Event Schema payload without enrichment. In enriched mode, it produces the full enriched payload.

This is the richest format in both modes and the only format that preserves all CI/CDecoy-specific fields including `_adapter` metadata.

Suitable for targets that accept structured JSON natively: Splunk HEC, Elasticsearch, webhooks, and log aggregators like Loki or Datadog.

## 7. NATS Consumption

### 7.1 Consumer Behavior

The forwarder creates up to two independent pull consumer loops depending on mode and configuration:

**Event consumer (always active):** Subscribes to `CONSUMER_STREAM` via the durable consumer named in `CONSUMER_NAME`. In enriched mode, this is the `ENRICHED_EVENTS` stream (post-pipeline). In normalized mode, this is the `DECOY_EVENTS` stream (post-adapter).

**Alert consumer (conditional):** Subscribes to `ALERT_STREAM` via the durable consumer named in `ALERT_CONSUMER`. This loop is active when both `ALERT_STREAM` and `ALERT_CONSUMER` are configured. In enriched mode, these are required â€” the ALERTS stream is a core output of the CTI pipeline. In normalized mode, these are optional â€” omit both variables to disable the alert loop entirely.

Both consumers operate independently when active. If the alert consumer stalls, event forwarding continues and vice versa.

### 7.2 Pull Subscription Loop

Each consumer loop operates as follows:

```
loop:
    messages = pull(batch_size=CONSUMER_BATCH_SIZE, wait=CONSUMER_BATCH_WAIT_MS)
    for each message:
        event = deserialize(message.data)
        payload = formatter.format(event)
        result = output.send(payload)
        if result.success:
            message.ack()
        else:
            message.nak(delay=backoff(message.num_delivered))
```

The deserialization step is identical in both modes â€” the forwarder parses the JSON payload into a struct that has optional enrichment fields. The formatter then checks field presence when producing output.

### 7.3 Acknowledgment Rules

| Scenario | Action | Rationale |
|---|---|---|
| Formatted and delivered successfully | `ACK` | Event is in the SIEM. |
| SIEM returned a retriable error (5xx, timeout, connection refused) | `NAK` with backoff delay | JetStream redelivers. Transient failures resolve. |
| SIEM returned a permanent error (4xx auth failure, 400 bad request) | `ACK` + log at `error` level | NAK would cause infinite redelivery. Log the dropped event for operator review. |
| Deserialization failure (malformed message on NATS) | `ACK` + log at `error` level | Cannot be fixed by redelivery. |
| Formatter error (event missing required fields for target format) | `ACK` + log at `warn` level with the event_id | Adapter-level bug. Don't poison the consumer. |

These rules are identical in both modes.

### 7.4 Backoff Strategy

NAK delay uses truncated exponential backoff: `min(2^(delivery_count - 1) * 1s, 60s)`.

| Delivery Attempt | NAK Delay |
|---|---|
| 1st | 1s |
| 2nd | 2s |
| 3rd | 4s |
| 4th | 8s |
| 5th+ | 60s (capped) |

The durable consumer's `max_deliver` setting (configured by the platform operator, not the forwarder) determines when a message is moved to the dead letter subject after exhausting retries.

## 8. Output Modules

Each output module implements a common interface. Output modules are mode-agnostic â€” they receive formatted byte payloads and deliver them. They do not inspect event content.

```go
type Output interface {
    // Name returns the output type identifier.
    Name() string

    // Send delivers a batch of formatted payloads to the SIEM target.
    // Returns per-message results so the consumer loop can ACK/NAK individually.
    Send(ctx context.Context, batch []FormattedEvent) []SendResult

    // HealthCheck verifies connectivity to the SIEM target.
    HealthCheck(ctx context.Context) error

    // Close flushes any pending data and releases resources.
    Close() error
}

type FormattedEvent struct {
    EventID   string    // For correlation with ACK/NAK
    Payload   []byte    // Formatter output
    Timestamp time.Time // For syslog timestamp, Splunk time field, etc.
    IsAlert   bool      // Allows output modules to route alerts differently
}

type SendResult struct {
    EventID string
    Success bool
    Err     error
    Retry   bool // True for transient errors, false for permanent
}
```

### 8.1 Syslog

Delivers events as RFC 5424 formatted syslog messages over TCP, UDP, or TCP+TLS.

**Connection management:** Maintains a persistent TCP connection to `SIEM_ENDPOINT`. Reconnects with exponential backoff on connection loss. UDP is fire-and-forget (always returns success).

**Message format:**

```
<{priority}>{version} {timestamp} {hostname} {app_name} {procid} {msgid} {structured_data} {msg}
```

| Field | Value |
|---|---|
| Priority | Facility (default `local0` = 128) + severity (see Â§6.6) |
| Version | `1` |
| Timestamp | Event `timestamp` in RFC 3339 format |
| Hostname | Forwarder pod hostname |
| App-Name | `SYSLOG_APP_NAME` (default `cicdecoy`) |
| ProcID | Forwarder PID |
| MsgID | `event_type` |
| Structured Data | `[cicdecoy decoy="{decoy_name}" tier="{tier}" session="{session_id}" mode="{forwarder_mode}"]` |
| Msg | The formatted payload (CEF, LEEF, JSON, or ECS depending on `OUTPUT_FORMAT`) |

**TLS:** When `SIEM_PROTOCOL=tcp+tls`, the forwarder validates the server certificate against the system CA pool or `SYSLOG_TLS_CA` if provided. Mutual TLS is supported via `SYSLOG_TLS_CERT` and `SYSLOG_TLS_KEY`.

### 8.2 Splunk HEC

Delivers events to the Splunk HTTP Event Collector endpoint.

**Endpoint:** `SIEM_ENDPOINT` should be the full HEC URL (e.g., `https://splunk.corp.internal:8088/services/collector/event`). If the path is omitted, `/services/collector/event` is appended automatically.

**Batching:** Events are batched into a single HTTP request using Splunk's newline-delimited JSON format. Batch flushes on `CONSUMER_BATCH_SIZE` messages or `CONSUMER_BATCH_WAIT_MS` timeout.

**Payload envelope:**

```json
{
  "time": 1743623253.219,
  "host": "bastion-dmz-01",
  "source": "cicdecoy",
  "sourcetype": "cicdecoy:event",
  "index": "cicdecoy",
  "event": { ... formatted event payload ... }
}
```

| Envelope Field | Value |
|---|---|
| `time` | Event `timestamp` as Unix epoch with millisecond precision |
| `host` | `source.decoy` value |
| `source` | `"cicdecoy"` |
| `sourcetype` | `SPLUNK_SOURCETYPE` for events, `SPLUNK_ALERT_SOURCETYPE` for alerts |
| `index` | `SPLUNK_INDEX` |
| `event` | Formatted payload (JSON or CEF string) |

**Authentication:** `Authorization: Splunk {SPLUNK_HEC_TOKEN}` header.

**Error handling:** HTTP 200 with `"code": 0` is success. HTTP 503 or connection failure is retriable. HTTP 401/403 is permanent (bad token). HTTP 400 is permanent (malformed event).

### 8.3 Elasticsearch

Delivers events using the Elasticsearch Bulk API.

**Endpoint:** `SIEM_ENDPOINT` should be the cluster base URL (e.g., `https://elastic.corp.internal:9200`). The forwarder appends `/_bulk` automatically.

**Index naming:** `{ELASTIC_INDEX}-{YYYY.MM.DD}` derived from each event's timestamp. For datastreams, the forwarder uses `op_type: "create"` in the bulk action.

**Batching:** Events accumulate in a bulk request body. Flush triggers on `ELASTIC_BULK_FLUSH_BYTES` payload size or `ELASTIC_BULK_FLUSH_INTERVAL_MS` timeout, whichever comes first.

**Bulk request format:**

```
{"create":{"_index":"cicdecoy-2026.04.02"}}
{ ... ECS-formatted event ... }
{"create":{"_index":"cicdecoy-2026.04.02"}}
{ ... ECS-formatted event ... }
```

**Authentication:** API key via `Authorization: ApiKey {ELASTIC_API_KEY}`, or basic auth via `ELASTIC_USERNAME` / `ELASTIC_PASSWORD`.

**Error handling:** Bulk API responses include per-item status. Items with `201` are successful (ACK). Items with `429` (rate limited) or `503` are retriable (NAK). Items with `400` are permanent failures (ACK + log). The forwarder parses each item in the response array and maps results back to individual NATS messages.

### 8.4 Webhook

Generic HTTP output for SOAR platforms, Slack/Teams integrations, PagerDuty, or custom consumers.

**Delivery:** Each event is sent as an individual HTTP request. Batching is not performed â€” webhook consumers vary too widely in what they accept.

**Request format:**

```
POST {SIEM_ENDPOINT}
Content-Type: application/json
{...additional headers from WEBHOOK_HEADERS...}

{ ... formatted event payload ... }
```

**Error handling:** HTTP 2xx is success. HTTP 429 or 5xx is retriable. HTTP 4xx (other than 429) is permanent.

**Timeout:** `WEBHOOK_TIMEOUT_MS` per request. Timeout is treated as retriable.

## 11. Lifecycle

### 11.1 Startup Sequence

1. Read and validate all environment variables. Validate that `FORWARDER_MODE` is `normalized` or `enriched`. In enriched mode, validate that `ALERT_STREAM` and `ALERT_CONSUMER` are set. Exit with code 1 and a descriptive error if validation fails.
2. Log the active configuration: mode, streams, output format, SIEM type, endpoint.
3. Initialize the formatter based on `OUTPUT_FORMAT`.
4. Initialize the output module based on `SIEM_TYPE`. Verify connectivity to the target (run `HealthCheck`). Log a warning if unreachable but do not exit â€” the target may come up later.
5. Connect to NATS. Retry with exponential backoff if unavailable.
6. Bind to the event consumer on `CONSUMER_STREAM`.
7. If `ALERT_STREAM` is configured, bind to the alert consumer.
8. Start serving `/healthz` and `/metrics`.
9. Begin the pull subscription loop(s).

### 11.2 Shutdown Sequence

1. Receive SIGTERM.
2. Stop pulling new messages from NATS on all active consumer loops.
3. Finish delivering any in-flight batches (10s timeout).
4. Flush the output module (e.g., Elasticsearch bulk buffer).
5. Close NATS connection.
6. Exit with code 0.
