# CI/CDecoy Adapter Contract Specification

**Version:** 1.0.0-draft
**Status:** Draft
**Last Updated:** 2026-04-02

---

## 1. Overview

An **adapter** is the boundary between a honeypot's native output and the CI/CDecoy telemetry pipeline. Its job is narrow by design: read raw events from a honeypot, translate them into the CI/CDecoy Common Event Schema, and publish them to NATS. Nothing else.

Adapters do not perform enrichment, correlation, ATT&CK mapping, GeoIP lookups, or alerting. Those responsibilities belong to downstream pipeline services that operate on the common schema. This constraint exists so that adapters remain thin, easy to write, and independently testable.

Any honeypot that produces structured or semi-structured output can be integrated into CI/CDecoy by writing an adapter that conforms to this contract.

---

## 2. Contract Summary

An adapter MUST:

1. Implement the `adapter.Adapter` interface (Go) or conform to the behavioral contract (any language).
2. Read from the honeypot's native output mechanism (log files, stdout, API, message queue).
3. Translate each raw event into a valid Common Event Schema payload.
4. Publish each translated event to NATS on the correct subject.
5. Prefix session IDs with the adapter name to avoid cross-adapter collisions.
6. Set a baseline `severity` value per event.
7. Expose a health check endpoint or signal.
8. Include `_adapter` metadata in every published event.

An adapter MUST NOT:

1. Perform MITRE ATT&CK technique mapping.
2. Perform GeoIP or ASN lookups on source IPs.
3. Correlate events across sessions or decoys.
4. Filter or drop events based on threat assessment.
5. Modify the NATS subject hierarchy beyond what this spec defines.
6. Make outbound network calls other than publishing to NATS.

## 3. Interface

### 3.1 Go Interface (Reference)

```go
package adapter

import (
    "context"
    "github.com/cicdecoy/cicdecoy/pkg/schema"
)

// Adapter defines the contract for translating honeypot output
// into CI/CDecoy common events.
type Adapter interface {
    // Name returns a stable, lowercase identifier for this adapter.
    // Used in NATS subjects, session ID prefixes, and telemetry.
    // Examples: "cowrie", "dionaea", "conpot", "custom-ssh"
    Name() string

    // Start begins reading from the honeypot source and sending
    // translated events to the provided channel. It blocks until
    // ctx is cancelled or a fatal error occurs. Transient read
    // errors (e.g., log rotation) should be retried internally.
    Start(ctx context.Context, events chan<- schema.Event) error

    // HealthCheck returns nil if the adapter can currently read
    // from its source. Called periodically by the runtime.
    HealthCheck(ctx context.Context) error
}
```

### 3.2 Behavioral Contract (Non-Go Implementations)

Adapters written in other languages must exhibit equivalent behavior:

- **Startup:** Begin reading from the honeypot source. Emit events as JSON payloads to NATS.
- **Shutdown:** On SIGTERM, drain in-flight events (up to 5s grace period), then exit cleanly.
- **Health:** Respond to HTTP GET on `/healthz` with `200 OK` when healthy, `503` otherwise.
- **Configuration:** Accept all configuration through environment variables (see §7).

## 4. Common Event Schema

Every event published to NATS MUST conform to this schema. Fields marked **required** must always be present. Fields marked **optional** may be omitted (not set to null).

### 4.1 Top-Level Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `event_id` | string (UUID v4) | yes | Unique identifier generated by the adapter per event. |
| `timestamp` | string (ISO 8601) | yes | Time the event occurred, as reported by the honeypot. If the honeypot does not provide timestamps, use the adapter's receive time. |
| `version` | string | yes | Schema version. Currently `"1.0"`. |
| `source` | object | yes | Identifies the originating decoy. See §4.2. |
| `session_id` | string | yes | Adapter-prefixed session identifier. Format: `{adapter_name}-{honeypot_session_id}`. Use `"none"` for connectionless events. |
| `event_type` | string | yes | One of the canonical event types defined in §5. |
| `source_ip` | string | yes | Attacker's IP address as observed by the honeypot. |
| `source_port` | integer | optional | Attacker's source port. |
| `severity` | string | yes | Baseline severity: `"info"`, `"low"`, `"medium"`, `"high"`, `"critical"`. |
| `data` | object | yes | Event-type-specific payload. See §5 for per-type field requirements. |
| `_adapter` | object | yes | Adapter provenance metadata. See §4.3. |

### 4.2 `source` Object

| Field | Type | Required | Description |
|---|---|---|---|
| `decoy` | string | yes | Decoy name as declared in the decoy manifest. Passed to the adapter via `CICDECOY_DECOY_NAME`. |
| `tier` | integer | yes | Interaction tier (1, 2, or 3) as declared in the decoy manifest. Passed via `CICDECOY_DECOY_TIER`. |
| `pod` | string | optional | Kubernetes pod name. Populated automatically from `HOSTNAME` if running in k8s. |
| `node` | string | optional | Kubernetes node name. Populated from `CICDECOY_NODE_NAME` if available. |

### 4.3 `_adapter` Object

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | yes | Adapter name, matching the return value of `Name()`. |
| `version` | string | yes | Adapter version (semver). |
| `original_event_type` | string | optional | The honeypot's native event type string before translation. Useful for debugging and auditing. |
| `ingest_latency_ms` | integer | optional | Milliseconds between the honeypot writing the event and the adapter publishing it. |

### 4.4 Example: Complete Event

```json
{
  "event_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "timestamp": "2026-04-02T18:47:33.219Z",
  "version": "1.0",
  "source": {
    "decoy": "bastion-dmz-01",
    "tier": 3,
    "pod": "decoy-bastion-dmz-01-7f8b9c-x4k2",
    "node": "k3s-worker-02"
  },
  "session_id": "cowrie-a1b2c3d4",
  "event_type": "command.exec",
  "source_ip": "45.33.32.156",
  "source_port": 48932,
  "severity": "medium",
  "data": {
    "command": "cat /etc/passwd",
    "cwd": "/home/admin",
    "username": "admin"
  },
  "_adapter": {
    "name": "cowrie",
    "version": "0.2.0",
    "original_event_type": "cowrie.command.input",
    "ingest_latency_ms": 12
  }
}
```

## 5. Event Types

Adapters translate honeypot-native events into one of the following canonical types. If a honeypot event does not map to any canonical type, the adapter SHOULD map it to the closest match and preserve the original type in `_adapter.original_event_type`. If no reasonable mapping exists, use `"unknown"` and include all available data in the `data` object.

### 5.1 `connection`

A new inbound connection to the decoy was established.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `protocol` | string | yes | Transport protocol: `"tcp"`, `"udp"`. |
| `service` | string | yes | Application protocol: `"ssh"`, `"http"`, `"smb"`, `"ftp"`, `"telnet"`, etc. |
| `dest_port` | integer | yes | Destination port the attacker connected to. |

**Severity guidance:** `"info"`

### 5.2 `auth.attempt`

A login or authentication attempt occurred.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `username` | string | yes | Username attempted. |
| `password` | string | optional | Password attempted. Omit if the honeypot does not capture it. |
| `method` | string | yes | Auth method: `"password"`, `"publickey"`, `"keyboard-interactive"`, `"ntlm"`, etc. |
| `accepted` | boolean | yes | Whether the decoy accepted the attempt. |
| `reason` | string | optional | Why accepted/rejected: `"realistic_accept"`, `"invalid_credentials"`, `"rate_limited"`, etc. |
| `attempt_number` | integer | optional | Sequential attempt count within the session. |

**Severity guidance:** `"low"` for single failures, `"medium"` for successful logins.

### 5.3 `command.exec`

A command was executed within an interactive session.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `command` | string | yes | The full command string as entered. |
| `cwd` | string | optional | Working directory at time of execution. |
| `username` | string | optional | User context the command ran under. |
| `uid` | integer | optional | UID if available. |
| `command_index` | integer | optional | Sequential command number within the session. |
| `env_snapshot` | object | optional | Key-value pairs of relevant environment variables. |

**Severity guidance:** `"medium"` baseline. Adapters MAY elevate to `"high"` for commands that are unambiguously malicious in context (e.g., downloading a remote payload), but SHOULD err toward letting the enrichment layer make that determination.

### 5.4 `file.access`

A file was read, written, uploaded, or downloaded.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `action` | string | yes | One of: `"read"`, `"write"`, `"upload"`, `"download"`, `"delete"`. |
| `path` | string | yes | File path within the decoy filesystem. |
| `size_bytes` | integer | optional | File size in bytes. |
| `sha256` | string | optional | SHA-256 hash of file contents, if available. |
| `username` | string | optional | User context. |

**Severity guidance:** `"medium"` for reads, `"high"` for uploads and writes (potential malware staging).

### 5.5 `honeytoken.triggered`

A honeytoken (canary credential, canary file, canary DNS record, etc.) was accessed.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `token_name` | string | yes | Honeytoken identifier as declared in the decoy manifest. |
| `token_type` | string | yes | Type of token: `"aws-credential"`, `"api-key"`, `"file"`, `"dns"`, `"url"`, etc. |
| `access_type` | string | yes | How it was accessed: `"file_read"`, `"api_call"`, `"dns_lookup"`, `"http_request"`, etc. |
| `accessed_path` | string | optional | Filesystem path or URL where the token was encountered. |
| `full_content_read` | boolean | optional | Whether the entire token value was read. |

**Severity guidance:** `"high"` or `"critical"`. Any honeytoken interaction is inherently suspicious.

### 5.6 `session.closed`

An interactive session ended.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `duration_seconds` | number | yes | Total session duration. |
| `commands_executed` | integer | optional | Total command count during the session. |
| `bytes_sent` | integer | optional | Bytes sent by attacker. |
| `bytes_received` | integer | optional | Bytes sent to attacker. |
| `close_reason` | string | optional | Why the session ended: `"client_disconnect"`, `"timeout"`, `"kicked"`, etc. |

**Severity guidance:** `"info"`

### 5.7 `alert`

A high-confidence indicator generated by the honeypot itself (not by the adapter's judgment). Only use this type if the honeypot has native alerting logic that identifies specific threat behaviors.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `alert_type` | string | yes | Free-form alert category from the honeypot. |
| `description` | string | yes | Human-readable description. |
| `related_event_ids` | array of strings | optional | Event IDs that contributed to this alert. |

**Severity guidance:** `"high"` or `"critical"`

### 5.8 `unknown`

A honeypot event that does not map to any canonical type. Adapters SHOULD avoid this where possible, but it is preferable to dropping events silently.

| `data` Field | Type | Required | Description |
|---|---|---|---|
| `raw` | object or string | yes | The original event data, preserved as-is. |

**Severity guidance:** `"info"`

## 6. NATS Publishing

### 6.1 Subject Format

```
cicdecoy.decoy.events.{decoy_name}.{event_type}
```

- `{decoy_name}` — The decoy's name as configured in `CICDECOY_DECOY_NAME`. Must be lowercase, alphanumeric with hyphens only. No dots.
- `{event_type}` — The canonical event type from §5, with dots replaced by underscores in the subject token. For example, `auth.attempt` becomes `auth_attempt` in the subject, and `file.access` becomes `file_access`.

**Examples:**

| Event | Subject |
|---|---|
| SSH connection to bastion-dmz-01 | `cicdecoy.decoy.events.bastion-dmz-01.connection` |
| Auth attempt on bastion-dmz-01 | `cicdecoy.decoy.events.bastion-dmz-01.auth_attempt` |
| Command exec on web-honeypot-03 | `cicdecoy.decoy.events.web-honeypot-03.command_exec` |
| Honeytoken triggered | `cicdecoy.decoy.events.bastion-dmz-01.honeytoken_triggered` |

### 6.2 Publishing Requirements

- Publish to NATS JetStream on the `EVENTS` stream.
- Messages MUST be JSON-encoded UTF-8.
- Messages MUST include a `Nats-Msg-Id` header set to the `event_id` for idempotent deduplication.
- If NATS is unreachable, the adapter MUST buffer events in memory (up to `CICDECOY_BUFFER_MAX_EVENTS`, default 10,000) and retry with exponential backoff.
- If the buffer is full, the adapter MUST log a warning and drop the oldest events. It MUST NOT block the read loop.
- On successful reconnection, the adapter MUST drain the buffer before resuming normal publishing.

### 6.3 JetStream Configuration (Reference)

Adapters do not create streams — the platform operator provisions them. This is the expected stream configuration adapters publish into:

```yaml
stream: EVENTS
subjects:
  - "cicdecoy.decoy.events.>"
retention: limits
max_bytes: 10737418240  # 10 GB
max_age: 604800000000000  # 7 days (nanoseconds)
storage: file
discard: old
duplicate_window: 120000000000  # 2 minutes (nanoseconds)
```

## 7. Configuration

All adapter configuration is provided through environment variables. Adapters MUST NOT read from configuration files unless the variable points to one (e.g., for complex honeypot-specific settings).

### 7.1 Required Variables

| Variable | Description | Example |
|---|---|---|
| `CICDECOY_DECOY_NAME` | Decoy name from the manifest. | `bastion-dmz-01` |
| `CICDECOY_DECOY_TIER` | Interaction tier (1, 2, 3). | `3` |
| `CICDECOY_NATS_URL` | NATS server connection URL. | `nats://nats.cicdecoy.svc:4222` |

### 7.2 Optional Variables

| Variable | Default | Description |
|---|---|---|
| `CICDECOY_NATS_CREDS_FILE` | (none) | Path to NATS credentials file for auth. |
| `CICDECOY_BUFFER_MAX_EVENTS` | `10000` | Max events buffered during NATS outage. |
| `CICDECOY_LOG_LEVEL` | `info` | Adapter log verbosity: `debug`, `info`, `warn`, `error`. |
| `CICDECOY_HEALTH_PORT` | `8080` | Port for the `/healthz` HTTP endpoint. |
| `CICDECOY_NODE_NAME` | (none) | Kubernetes node name, set via downward API. |

### 7.3 Adapter-Specific Variables

Adapters may define additional variables prefixed with their name in uppercase. These must be documented in the adapter's own README.

| Variable | Adapter | Description |
|---|---|---|
| `COWRIE_LOG_PATH` | cowrie | Path to Cowrie's JSON log directory. Default: `/var/log/cowrie`. |
| `DIONAEA_SQLITE_PATH` | dionaea | Path to Dionaea's SQLite log database. |
| `TPOT_ES_URL` | tpot | Elasticsearch URL for T-Pot's data store. |

## 8. Deployment Patterns

### 8.1 Sidecar (Recommended)

The adapter runs as a sidecar container in the same pod as the honeypot. They share a log volume.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: decoy-bastion-dmz-01
  namespace: cicdecoy
spec:
  containers:
    - name: cowrie
      image: ghcr.io/csquare-d/cowrie:latest
      volumeMounts:
        - name: logs
          mountPath: /var/log/cowrie

    - name: adapter
      image: ghcr.io/csquare-d/adapter-cowrie:latest
      env:
        - name: CICDECOY_DECOY_NAME
          value: "bastion-dmz-01"
        - name: CICDECOY_DECOY_TIER
          value: "3"
        - name: CICDECOY_NATS_URL
          value: "nats://nats.cicdecoy.svc:4222"
        - name: COWRIE_LOG_PATH
          value: "/var/log/cowrie"
      volumeMounts:
        - name: logs
          mountPath: /var/log/cowrie
          readOnly: true
      livenessProbe:
        httpGet:
          path: /healthz
          port: 8080
        periodSeconds: 30

  volumes:
    - name: logs
      emptyDir: {}
```

### 8.2 Standalone

For honeypots that expose output over a network interface (Elasticsearch, API, syslog), the adapter runs as a separate deployment.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: adapter-tpot
  namespace: cicdecoy
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: adapter
          image: ghcr.io/csquare-d/adapter-tpot:latest
          env:
            - name: CICDECOY_DECOY_NAME
              value: "tpot-external-01"
            - name: CICDECOY_DECOY_TIER
              value: "1"
            - name: CICDECOY_NATS_URL
              value: "nats://nats.cicdecoy.svc:4222"
            - name: TPOT_ES_URL
              value: "http://tpot-host:64298"
```

## 9. Testing Requirements

Adapter authors MUST provide:

### 9.1 Schema Validation Tests

Every event the adapter can produce must be validated against the Common Event Schema JSON Schema (published at `pkg/schema/event.schema.json`). The test suite must cover all event types the adapter supports.

### 9.2 Translation Fidelity Tests

For each supported honeypot event type, provide at least one fixture file containing the honeypot's raw output and the expected translated event. The test asserts field-by-field equality.

```
adapters/cowrie/testdata/
├── raw/
│   ├── login_attempt.json
│   ├── command_input.json
│   └── session_closed.json
└── expected/
    ├── login_attempt.json
    ├── command_input.json
    └── session_closed.json
```

### 9.3 Integration Smoke Test

A Docker Compose file that starts the honeypot, the adapter, and a NATS server. A test script sends a synthetic interaction to the honeypot and asserts that a valid event appears on the expected NATS subject within 10 seconds.

```
adapters/cowrie/test/
├── docker-compose.yml
├── trigger.sh       # Sends a test SSH connection to Cowrie
└── verify.sh        # Subscribes to NATS subject and validates output
```

## 10. Adapter Lifecycle

### 10.1 Startup Sequence

1. Read configuration from environment variables.
2. Validate required variables are present. Exit with code 1 and a clear error message if any are missing.
3. Connect to NATS. If unavailable, retry with exponential backoff (max 60s interval). Do not exit.
4. Begin reading from the honeypot source.
5. Start serving `/healthz` on the configured port.

### 10.2 Shutdown Sequence

1. Receive SIGTERM.
2. Stop reading new events from the honeypot source.
3. Drain any buffered events to NATS (5s timeout).
4. Close NATS connection.
5. Exit with code 0.

### 10.3 Error Handling

| Scenario | Behavior |
|---|---|
| Honeypot log file rotated | Re-open and continue. Do not crash. |
| Honeypot log file missing at startup | Wait and retry. Log a warning every 30s. |
| Malformed honeypot event | Log the raw event at `warn` level, skip it, continue. |
| NATS connection lost | Buffer events in memory, reconnect with backoff. |
| Buffer full during NATS outage | Drop oldest events, log a warning with the count of dropped events. |
| Unrecoverable error | Log at `error` level, exit with code 1. The orchestrator restarts the container. |

## 11. Contributing a New Adapter

### Step-by-step

1. Run `cicdecoy scaffold adapter --name my-honeypot` to generate a boilerplate directory.
2. Implement the `Adapter` interface in `adapters/my-honeypot/my-honeypot.go`.
3. Add fixture files in `adapters/my-honeypot/testdata/`.
4. Write schema validation and translation fidelity tests.
5. Create a Docker Compose integration test.
6. Document adapter-specific environment variables in `adapters/my-honeypot/README.md`.
7. Add a row to the adapter compatibility matrix below.
8. Submit a pull request.

### Adapter Compatibility Matrix

| Adapter | Honeypot | Event Types Supported | Interaction Level | Status |
|---|---|---|---|---|
| `cowrie` | Cowrie | connection, auth.attempt, command.exec, file.access, session.closed | High | Reference |
| `dionaea` | Dionaea | connection, auth.attempt, file.access | Medium | Planned |
| `tpot` | T-Pot (via ES) | connection, auth.attempt, command.exec, file.access | Varies | Planned |
| `conpot` | Conpot | connection, auth.attempt | Low | Planned |

## Appendix A: Severity Assignment Guide

Adapters assign a **baseline** severity. Downstream enrichment services may elevate it based on context (e.g., known-malicious source IP, ATT&CK technique correlation). Adapters SHOULD NOT lower severity below these baselines.

| Signal | Severity |
|---|---|
| Connection established | `info` |
| Single auth failure | `low` |
| Successful auth on decoy | `medium` |
| Command execution | `medium` |
| File read | `medium` |
| File write / upload | `high` |
| Honeytoken access | `high` or `critical` |
| Honeypot-native alert | `high` or `critical` |
| Session ended normally | `info` |

## Appendix B: Session ID Prefixing

Session IDs MUST be prefixed to guarantee uniqueness across adapters. Two honeypots in the same cluster could independently generate identical session IDs. The prefix eliminates collisions.

**Format:** `{adapter_name}-{honeypot_native_session_id}`

**Examples:**
- Cowrie session `a1b2c3d4` → `cowrie-a1b2c3d4`
- Dionaea session `12345` → `dionaea-12345`
- Custom adapter session `sess_001` → `custom-ssh-sess_001`

If the honeypot does not generate session IDs, the adapter MUST generate a UUID v4 and prefix it: `cowrie-{uuid}`.
